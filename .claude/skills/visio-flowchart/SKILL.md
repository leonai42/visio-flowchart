# Visio 流程图生成器

## 前置检查

在生成流程图之前，按顺序完成以下检查。任一步失败则停止执行并提示用户。

1. **uv 可用性**：检查 `uv` 是否已安装（执行 `uv --version`）。如果未安装 → 提示 `请先安装 uv：https://docs.astral.sh/uv/getting-started/installation/`，停止执行。
2. **Microsoft Visio 运行状态**：检查 Visio 是否已启动。如果未启动 → 提示 `请先启动 Microsoft Visio 后再试`，停止执行。
3. **visio-mcp 可用性**：尝试调用 `list_diagram_types`。如果失败 → 提示 `请先安装 visio-mcp：pip install visio-mcp`，停止执行。

## 工作流程

### 第一步：解析用户意图

从用户的自然语言输入中提取以下信息。

#### 1.1 流程图类型判断

| 输入包含关键词 | 流程图类型 | create_diagram 参数 |
|--------------|-----------|-------------------|
| 泳道 / swimlane / 跨职能 / 部门 / 角色 | 泳道图 | `"cross_functional_flowchart"` |
| 其他情况 | 基本流程图 | `"basic_flowchart"` |

#### 1.2 节点提取

1. 将用户输入按分隔符拆分为步骤序列。分隔符包括：`→`、`->`、`—`、`>`、`然后`、`接着`、`下一步`、`之后`、`再`
2. 每个步骤作为一个节点，保持原有顺序
3. 对每个节点进行类型判定（见下表）

**节点类型判定规则**：

| 节点文本特征 | 节点类型 |
|------------|---------|
| 第一个步骤（无前置） | 开始 |
| 最后一个步骤（无后续） | 结束 |
| 包含"是否/如果/判断/验证…？/审批…？/通过？/成功？/正确？/存在？" | 判断 |
| 包含"输入/输出/读取/写入/保存/加载/数据/表单" | 数据 |
| 包含"子流程/子模块/详见/展开" | 子流程 |
| 其他 | 处理 |

#### 1.3 分支解析

当用户描述中包含条件分支时（如"通过则…，不通过则…"、"是则…，否则…"）：

- 判断节点有两个出口：`是` 分支（通过路径）和 `否` 分支（不通过路径）
- 分别提取两条路径上的后续节点
- 两条路径最终汇合到同一节点，或分别结束

#### 1.4 泳道角色提取（仅泳道图）

从用户描述中提取执行主体关键词作为泳道角色：
- 通用角色关键词：`用户`、`客户`、`管理员`、`操作员`
- 部门角色关键词：`财务`、`人事`、`经理`、`主管`、`部门`
- 系统角色关键词：`系统`、`服务端`、`客户端`、`前端`、`后端`、`数据库`
- 业务角色关键词：`仓库`、`物流`、`供应商`、`采购`、`销售`
- 泳道按角色在描述中的首次出现顺序从上到下排列

---

### 第二步：判断复杂度

1. 计算节点总数：**主路径节点数 + 所有分支路径节点数（含判断节点自身）**
2. 应用阈值规则：

| 节点总数 | 交互模式 |
|---------|---------|
| ≤ 5 | **直接生成**：跳过确认步骤，直接进入第三步 |
| > 5 | **先确认再生成**：展示节点清单，等待用户确认 |

#### 确认清单格式（>5 节点时展示）

```
📋 流程图节点清单（共 N 个节点）：

  1. [开始]  <节点名称>                       (圆角矩形)
  2. [处理]  <节点名称>                       (矩形)
  3. [判断]  <节点名称>                       (菱形)
     ├─ 是 → 4
     └─ 否 → 5
  4. [处理]  <节点名称>                       (矩形)
  5. [处理]  <节点名称>                       (矩形)
  N. [结束]  <节点名称>                       (圆角矩形)

请确认以上节点：
- 回复"确认"或"ok"开始生成
- 或直接描述需要修改的地方（如"增加一个通知节点"、"把审核改成两级审核"）
```

#### 用户响应处理

- 用户回复"确认"/"ok"/"没问题"/"可以" → 以确认的清单进入第三步
- 用户回复包含修改意见 → 更新节点清单，重新展示确认格式
- 用户回复"取消" → 停止执行

---

### 第三步：映射形状规范

#### 3.1 读取公司规范配置

首先尝试读取 `.claude/skills/visio-flowchart/standards.md` 文件。

- **文件存在**：解析配置表中的覆盖项。对每项进行格式校验（见 3.3），有效的覆盖 ISO 5807 默认值，无效的忽略并提示用户。
- **文件不存在**：全部使用 ISO 5807 默认值，不报错。

#### 3.2 ISO 5807 默认形状映射

| 节点类型 | Visio 形状 | Master 名称参考 |
|---------|-----------|---------------|
| 开始/结束 | 圆角矩形 | `Rounded Rectangle` / `Start/End` |
| 处理 | 矩形 | `Rectangle` / `Process` |
| 判断 | 菱形 | `Diamond` / `Decision` |
| 数据/输入输出 | 平行四边形 | `Parallelogram` / `Data` |
| 子流程 | 双边框矩形 | `Subprocess` / `Predefined Process` |

**默认样式值**（当 standards.md 无覆盖时）：

| 属性 | 默认值 |
|------|--------|
| 填充色 | `#FFFFFF`（白色） |
| 线条色 | `#333333`（深灰） |
| 字体 | `微软雅黑` |
| 字号 | `11pt` |
| 连接线箭头 | 实心三角箭头 |
| 连接线颜色 | `#333333` |

#### 3.3 配置值校验

- **颜色值**：必须匹配 `^#[0-9A-Fa-f]{6}$` 格式。不匹配 → 忽略该项，提示 `standards.md 中颜色值 "<值>" 格式无效，已使用默认值`
- **字体**：不作严格校验，直接传给 Visio。Visio 自身会在字体不可用时回退到系统默认字体。

#### 3.4 双层映射逻辑

对每个节点：
1. 查找 standards.md 中对应"节点类型"行的配置
2. 如果某项有有效值 → 使用配置值
3. 如果某项为空或无效 → 使用 ISO 5807 默认值
4. 形状类型、颜色、字体各自独立覆盖，互不影响

---

### 第四步：编排 MCP 调用

严格按以下顺序调用 visio-mcp 工具：

#### 4.1 create_diagram

调用 `create_diagram("<type>")`，其中 `<type>` 为第一步确定的流程图类型（`"basic_flowchart"` 或 `"cross_functional_flowchart"`）。

- 调用成功后会返回 diagram standard（包含该图表类型支持的 shapes、connectors、layout 规范）
- 将此返回值保存，用于后续步骤中的精确 master 名称和布局参数
- 如果返回的 standard 中缺少所需信息，可调用 `get_diagram_standard("<type>")` 补充查询
- 调用失败 → 停止执行，提示 `Visio 创建图表失败，请检查 Visio 是否正常运行`

#### 4.2 batch_draw_shapes

将所有节点一次性创建。调用失败 → 停止执行，提示 `形状创建失败，请检查 Visio 模板是否正确加载`。

**基本流程图布局**：
- 节点垂直排列，间距 1.5 英寸
- 主流程居中，分支节点放置在主流程右侧
- 节点宽度统一约 1.5 英寸，高度约 0.75 英寸
- 开始节点放在顶部，结束节点放在底部

**泳道图布局**：
- 先创建泳道分隔条（每个角色一条泳道）
- 泳道按角色首次出现顺序从上到下排列，间距 2.5 英寸
- 将节点放入对应角色的泳道中
- 同泳道内节点垂直排列，间距 1.5 英寸

使用 `get_diagram_standard` 返回的精确 master 名称。如果没有返回，使用第三步映射的常见 master 名称。

#### 4.3 batch_connect_shapes

创建所有连接线。调用失败 → 提示 `连接线创建失败，请检查形状 ID 是否正确`。

- 顺序节点间使用带箭头的直线连接器
- 判断节点 → 两个出口：`是` 标签（向右或向下）和 `否` 标签（向下或向左）
- 分支汇合点：如果两条路径汇合，两条线指向同一目标节点
- 泳道图：跨泳道连接线需跨越泳道分隔线，箭头指向目标节点所在泳道

#### 4.4 set_shape_format

对每个形状应用样式。单个形状设置失败 → 跳过该形状继续，最后汇总跳过的形状 ID 告知用户。

- 逐个形状调用 `set_shape_format`
- 传入第三步确定的：填充色（fill_color）、线条色（line_color）、字体（font_name）、字号（font_size）
- 判断节点的连接线需额外设置分支标签文本

#### 4.5 save_document_as

保存文件：

- 文件名：从用户输入中提取流程主题 + `.vsdx`
  - 示例：用户说"画个用户登录流程" → `用户登录流程.vsdx`
  - 无法提取主题时使用 `流程图.vsdx`
- 保存路径：优先使用当前 Claude Code 会话的工作目录。如果 MCP 环境导致路径不确定，先调用 `save_document_as` 不指定路径获取默认保存位置，再提示用户文件所在位置
- 如文件已存在 → 提示 `文件 "<文件名>" 已存在，是否覆盖？`，等待用户回复
- 保存失败（权限不足/磁盘空间不足）→ 提示 `文件保存失败，请检查磁盘空间和目录写入权限`

## 泳道图特别规则

- 泳道按角色首次出现顺序从上到下排列，间距 2.5 英寸
- 节点根据第二步提取的执行主体分配到对应泳道
- 无法确定归属的节点（如"开始"、"结束"）放在第一个泳道
- 跨泳道连接线穿越泳道边界时保持垂直方向，避免斜线交叉
- 在 `create_diagram` 时使用 `"cross_functional_flowchart"` 模板以获得泳道模具

## 常见模板

当用户描述与以下场景匹配时，优先使用预设节点清单作为起点，再根据用户具体需求调整（增删节点、修改分支）。

### 登录认证流程
```
[开始] → [输入账号密码] → [验证账号] → [验证通过?] → [进入首页] → [结束]
                                                   └─ 否 → [显示错误提示] → [重新输入]
```

### CRUD 操作流程
```
[开始] → [接收请求] → [校验参数] → [参数有效?] → [执行数据库操作] → [返回结果] → [结束]
                                              └─ 否 → [返回错误信息] → [结束]
```

### 审批流程
```
[开始] → [提交申请] → [直属上级审批] → [审批通过?] → [部门负责人审批] → [审批通过?] → [执行] → [结束]
                                                    └─ 否 → [退回修改] → [重新提交]
                               └─ 否 → [退回修改] → [重新提交]
```

### 订单处理流程
```
[开始] → [用户下单] → [系统验证库存] → [库存充足?] → [锁定库存] → [用户支付] → [支付成功?] → [仓库发货] → [用户收货] → [结束]
                                                 └─ 否 → [提示缺货] → [结束]
                                                                          └─ 否 → [取消订单] → [退款] → [结束]
```

### 异常处理流程
```
[开始] → [正常执行] → [发生异常?] ── 否 → [继续执行] → [结束]
                         │
                         是
                         ↓
                    [记录日志] → [判断异常类型] → [可恢复?] ── 是 → [重试] → [正常执行]
                                                         │
                                                         否
                                                         ↓
                                                    [降级处理] → [告警通知] → [返回错误] → [结束]
```

## 风格指南

生成流程图时遵循以下原则：
- 节点文本简洁，不超过 15 个字
- 判断节点文本以 `？` 结尾
- 开始节点统一用 `开始`，结束节点统一用 `结束`
- 主流程从上到下，分支从左到右（或从上到下，取决于空间）
- 避免连接线交叉，必要时调整节点位置
- 泳道标题使用角色名称，加粗显示
- 同一条路径上的节点保持相同宽度
